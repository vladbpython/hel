## [1.3.1]

### Fixed

- **`shard_group` batch send lost data.** When the receiver closed,
  `ShardGroup::send_batch_async` threw away the groups it had not gotten to yet.
  A batch covering several sectors could lose whole sectors, quietly, with no
  error. Every other keyed batch method already returned unsent items; this one
  did not.

- **`shard_group` batch send lost the whole batch when cancelled.** It kept the
  sorted batch inside the future instead of in your buffer, so dropping the
  future (under `select!`, `timeout`, or task cancellation) took everything with
  it. You were left holding an empty buffer and no error.

- **Batch sends now survive cancellation on every channel.** Drop the future at
  an `.await` and whatever has not been sent stays in your buffer, in order.
  This covers `Sender::send_batch_async`, `ShardKey::send_batch_async`,
  `ShardGroup::send_batch_async`, and `ShardRoundRobin::send_batch_async`. Before,
  the item being sent right then was always lost, and `Sender::send_batch_async`
  handed the buffer back reversed.

- **`try_send_batch` could hang on a full channel.** It promises not to block,
  but on macOS it could get stuck spinning until a consumer drained the queue.
  Blocking senders reserve their slot up front, which pushed the write position
  past what the batch code expected, and it kept retrying instead of giving up.
  It now returns right away. Linux was never affected.

- **An async send could wait forever if polled after it finished.** That only
  happens if the caller misuses the future, but hanging is a bad way to say so.
  It now just returns.

- **Keyed batch sends are faster when the channel is full.**
  `ShardKey::send_batch_async` used to sort the whole buffer again for every item
  it had to wait on, which got slow with big batches and a slow consumer. It now
  sorts once.

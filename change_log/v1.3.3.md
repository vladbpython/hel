## [1.3.3]

### Fixed

- **Lost wakeup in `SyncList::pop_async`.** If a notification arrived at the same time as a future was being cancelled, `pop_async` could pop a dying slot from the queue, see that the cancelled flag was not set yet while the waker was already taken, and give up. As a result, every live waiter behind that slot was left without a wakeup. Now an empty slot is handled the same way as a cancelled one: skip it and move on to the next. The bug was confirmed with Loom and pinned by the regression test `notify_never_starves_live_waiter`: reverting the fix makes the test fail. Verified.

- **Orphaned pool shards on handler panic.** If a worker panicked inside user handler code, it stayed the owner of its shards forever. Other workers could not take them over, and the shards silently stopped being processed. Every worker now holds an RAII `OwnerGuard` that releases its shards on every exit path, including panic unwind.

- **`claim_or_release` lost a successful claim.** A worker that won the CAS for a shard reported failure and skipped one processing pass for that shard. The CAS result is now returned directly.

- **`push_batch` could spin on reservation overshoot.** The state where `tail - head` is greater than the channel capacity is a normal situation while blocked senders hold their fetch_add reservations, not a transient snapshot error. The two cases are now told apart: a stale snapshot is re-read, a reservation overshoot returns zero right away.

- **`poll_send` after completion.** Polling an already completed send future now always answers `Ready(Ok(()))`, like a fused future, instead of behavior depending on internal state. No data can be lost this way: a future that finished with `Ready(Err(v))` has already returned the value inside the error.
- 
- **`SyncList::cancel_async_slot`** is now an instance method and performs lazy cleanup. Cancelled slots are removed from the head of the queue at the moment of cancellation. Thanks to this, the queue length is bounded by the number of live waiters, not by the history of cancellations. Cancelled slots in the middle of the queue are removed once they reach the head. FIFO order is preserved, and there is no expensive removal from the middle.

### Added

- **Model checking with Loom.** Loom exhaustively explores every possible thread interleaving for the cancel plus notify pair. Two tests:
    - `notify_never_starves_live_waiter` - regression for the lost wakeup, fails on the old code.
    - `concurrent_cancels_keep_count_consistent` - two concurrent cancels never drift from the real queue length.

- **Internal `shim::loom` type facade.** In a normal build it resolves to std, parking_lot and atomic-waker, exactly as before. Under `--cfg loom` the same places build on Loom instrumented types. Zero impact on regular builds; loom is not even downloaded as a dependency without the special flag.

### Changed

- **`Clone` for `Receiver` is now gated by the `MultiConsumer` marker trait.** The SPSC protocol allows exactly one consumer: `SingleInner::pop` advances `head` with a plain store, no CAS, so two consumers would race on `head` and could read the same slot twice. Previously this invariant was held by convention only — a `Receiver` backed by `SingleInner` could still be cloned inside the crate. Now `Clone` exists only for inners that explicitly implement `MultiConsumer` (`SeqInner` does, `SingleInner` does not), so a second consumer on the SPSC protocol is a compile error. This mirrors the existing `MultiProducer` marker on the sender side: both halves of the protocol invariant are now enforced at the type level.
  No behavior change for the public API: the SPSC types exposed through `channel::spsc` never offered `Clone`, and every cloneable receiver path goes through the MPMC inner, which implements the marker.

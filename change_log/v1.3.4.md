## [1.3.4]

### Added

**`send_ref_async`: cancel safe async send that never loses the value.**
A new way to send. The value stays in the caller's frame as
`&mut Option<T>` and the future only borrows it. The rule is simple and
the same for every outcome. If the call returns `Ok(())`, the slot is
empty and the value is in the channel. If the call returns
`Err(Disconnected)`, the value is still in the slot. If the future is
cancelled while waiting (for example it loses a `select!` race), the
value is also still in the slot. The owned `send_async` cannot give this
guarantee: on cancellation it drops the unsent value. Silent delivery on
cancellation is impossible here by design. The value enters the channel
only through a plain `push` inside `poll`, so a receiver can never take
it behind the future's back. One thing to know: an empty slot on entry
means "nothing to send" and the call completes with `Ok(())` right away.
If you refill the slot in a loop, refill it only when `slot.is_none()`,
otherwise you can overwrite a value that was cancelled and recovered.
There is no extra cost on the hot path. The future compiles down to the
same `poll_send` as `send_async`, and the owned path is not touched at
all. The trade off is a lifetime: the future borrows the sender and the
slot, so it cannot be moved into `tokio::spawn`. It is meant for
`select!` and `timeout` in the caller's own frame.
Pinned by the tests `cancel_while_full_keeps_value`,
`disconnect_keeps_value`, `success_clears_slot` and
`cancelled_sender_passes_baton`. The last one checks that cancelling a
parked sender wakes up the next waiting sender instead of losing the
wakeup.

**`send_ref_async` on every shard wrapper.**
The same contract is available on SPSC, MPMC round robin, `ShardGroup`
(by handle) and `ShardKey` (by key). The shard is picked before the
await, which gives each wrapper a clear cancellation behavior. Round
robin spends its rotation step even when the send is cancelled, so a
retry goes to the next shard. Round robin never promised any placement,
so this is fine, and the test
`rr_send_ref_async_cancel_consumes_rotation_tick` pins it. Group and key
routing is deterministic, so a retry lands on the same shard. Per key
order is kept as long as the caller retries the recovered value before
sending anything new for that key. The key wrapper builds the key
`String` only on the disconnect path. Cancellation allocates nothing.

**New error types `AsyncSendRefError`, `ShardAsyncSendRefError` and
`ShardKeyAsyncSendRefError`.**
Errors for the borrow based send. Unlike `AsyncSendError<T>` they do not
carry `T`, because the value is already in the caller's slot.

**Zero loss graceful shutdown examples.**
Four runnable examples (SPSC, round robin, key, group). Producers use
`send_ref_async` with a `CancellationToken` and a biased `select!`. They
get cancelled in the middle of sending, the in flight value is recovered
from the slot, and the `sent == received` count matches exactly. The
group example also checks a stronger property: the recovered value is
equal to the shard's receive count, which proves at the same time that
the value was not lost and was not silently delivered.

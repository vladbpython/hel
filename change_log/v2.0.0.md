## [2.0.0]

### Note

I ask for your forgiveness. After thorough testing of the `pool` feature, a refactoring became necessary, and batch handler dispatch is not supported at the moment. Safety and transparency come first: when a batch handler panics mid batch, there is no way to know how many items it already processed, so safely recovering N inflight items without duplicates or other side effects is not achievable. Every design we tried either lost data silently, risked processing items twice, or hid the weakened guarantee behind a configuration flag, and none of that belongs in this crate.

The pool now processes items one at a time. The blast radius of any panic, anywhere, is exactly one item, and with the slot pools that item is handed back to you together with the panic reason instead of disappearing.

If you need batch processing in the pool, or you have an idea for how to do it safely, please open an issue: **pool batch solution**. I would genuinely like to hear real use cases before designing this again.

### Breaking changes

- **Batch handler dispatch removed.** The `Batch` handler wrappers (sync and async) and the batched worker paths are gone. Handlers are now called one item per call, unconditionally. A panic costs at most one item, never a whole inflight batch.
- **Handler traits replaced by the slot contract.** `AsyncHandler` and `SyncHandler` are removed. The pool is built on `AsyncSlotHandler` and `SyncSlotHandler`:

  ```rust
  fn handle(&self, slot: &mut Option<T>);  // sync
  fn handle(&self, slot: &mut Option<T>) -> impl Future<Output = ()>; // async
  ```

  The worker owns the item. The handler reads it by reference and calls `slot.take()` only at its commit point. A panic before `take()` leaves the item with the worker, so it is recoverable. A panic after `take()` means the handler accepted ownership. This mirrors the cancel safe `send_ref_async` design on the sending side: the value stays in the owner's slot until committed.
- **Pool constructors.** `sync_pool` and `async_pool` are replaced by `sync_pool_slot` and `async_pool_slot`, both of which take a dead letter sink.
- **Dead letter sink signature.** The sink receives the item together with the reason it failed:

  ```rust
  D: Fn(T, PanicReason) + Send + Sync + 'static
  ```

- **Minimum channel capacity is now 2.** `CAP = 1` is rejected at construction. See Fixed below for why.
- Fix in the public API: `get_singal_stop` is now `get_signal_stop`.

### Added

- **Zero loss guarantee, proven by tests.** With the slot pools, every item is either processed exactly once or delivered to the dead letter sink exactly once. The test suite asserts this as an exact equality (`ok + dead_lettered == N`), including per key sum checks across `round_robin`, `shard_key`, and `shard_group` routing, for both sync and async pools.
- **`PanicReason`.** Wraps the raw panic payload. `message()` returns the human readable text for string panics, which covers `panic!`, `unwrap`, `expect`, asserts, index and arithmetic panics. `into_inner()` gives the raw payload for custom downcasting. `Result::unwrap` and `expect` payloads include the `Debug` form of the underlying error, so a dead lettered item arrives with the actual failure, not just the fact of one.
- **Panic accounting.** `pool.handler_panics()` exposes a counter of caught handler panics. Workers survive handler panics and keep their shards; a panicking dead letter sink cannot kill a worker either (the item is dropped in that case, but counted).
- **`PerItem` wrapper.** The single library handler wrapper, borrow based (`Fn(&T)` / `Fn(&T) -> Future`), implements both slot traits and always preserves the zero loss guarantee. Handlers that need ownership of `T` implement the slot trait directly, so the `take()` and the responsibility boundary it draws are visible in user code.
- **Loom verification suite.** Exhaustive interleaving models for the Vyukov ring (wrap around, contended tail, two phase batch publication versus pop, close/drain shutdown race, abort seal on receiver close, over reservation), the async waiter list (lost wakeup, cancellation races), and the pool's shard ownership protocol (mutual exclusion under scaling, guard release on unwind). Run with `RUSTFLAGS="--cfg loom" cargo test --lib loom --release --features pool`.
- **Miri coverage** for the rewritten ring internals: placement initialization, drop of undelivered items, abort holes in the sequence chain (including the two hole case), and panic unwind paths.

### Fixed

- **Silent overwrite at `CAP = 1`.** The sequence ring is algorithmically unsound at capacity 1: the "slot is free" condition (`seq == pos`) and the "slot was consumed" condition (`seq == pos + CAP`) collide, so a producer could overwrite an unread item without any error. Found by the loom model `algo_ring_1p1c_wraparound_exact`. Both ring constructors now reject `CAP < 2` with a clear message, backed by `should_panic` regression tests.
- Handler panics no longer kill workers, desynchronize the active consumer count, or starve the dead worker's shards.

### Performance

- The per iteration panic guard is free on the non panicking path (verified by A/B measurement).
- Per item dispatch costs up to 23 percent versus the removed batched dispatch in the worst case, measured with a no op handler; with any real per item work the difference disappears into the workload. Receive side batching (`batch_size`) is unchanged and still amortizes channel traffic.

### Migration from 1.x

1. Replace `sync_pool(...)` / `async_pool(...)` with `sync_pool_slot(...)` / `async_pool_slot(...)` and add a dead letter sink as the last argument. A minimal sink is `|_item, _panic_reason| {}`, but logging the cause is strongly recommended.
2. Replace `PerItem(move |v: T| ...)` closures with borrow form `PerItem(move |v: &T| ...)`.
3. Replace `Batch(move |batch: &[T]| ...)` handlers with `PerItem` and move aggregation into shared state (atomics for counters and sums), or buffer inside a custom slot handler. See the reworked examples for both patterns.
4. If you constructed channels with capacity 1, use 2 or more.
